# Meeting PPT Creator

从对话中提炼出的会议材料制作规范，适用于内部团队会议的幻灯片和配套文档。

## 核心原则

### 会议形式设计
- **议题分类**：先区分「讨论型」和「汇报型」议题。讨论型不给结论，给框架和问题引发参与；汇报型先讲结论再给支撑。
- **轻松开放的会议**：不把结论写死在 PPT 里，用开放性问题引导，结尾留「讨论方向」而非「行动结论」。
- **现状先于结论**：先摆事实和外部压力，让听众自己感受到问题，再给出结论方向。不要在第一页就告诉别人答案。
- **散会前对齐**：最后一页必须是具体的行动清单，含责任人和时间节点，会议结论落地。

### PPT 叙事逻辑
- **问题先于答案**：用现状描述 + 外部变量引出问题，再给结论，再展开支撑。
- **结论支撑充分**：每个结论背后有具体的数据、案例或逻辑链，不只是断言。
- **重点突出**：一页一个核心信息，次要信息降权显示（字体缩小、颜色变浅）。
- **语言简练**：去掉所有可删不影响意思的字，标题用动词短句，正文用短句或条目。
- **过渡页不可少**：当逻辑层次跨度大时，插入过渡页说明「从哪来、到哪去」，避免突兀跳转。

### 信息结构设计
- **上层框架 + 下层细节**：先用视觉突出的色块/图示展示核心框架（如三个层次、两类分组），再用细节卡片对应展示每项的具体内容。避免把框架和细节平铺在同一层级。
- **分组有依据**：五项能力/三个阶段/四类内容等并列项，要按内在逻辑分组，而非简单平铺，并用视觉分隔线或色彩区分不同组。
- **对应关系可视化**：当「考核手段对应考核目标」「环节对应能力」这类映射关系存在时，用列对齐或箭头图明确展示，不要让读者自己脑补对应关系。

## HTML 幻灯片规范

### 技术要求
- 纯 HTML + CSS + 少量 JS，单文件，无外部依赖（字体除外）
- 翻页控件固定在页面**右侧垂直居中**，纵向排列（上箭头 / 页码 / 下箭头），避免遮挡内容
- 支持键盘左右方向键翻页
- 每页右上角显示页码（当前页 / 总页数）

### 视觉风格
- 亮色商务风，白底为主，彩色 accent 区分信息层次
- 顶部 4px 色条区分议题（蓝色=议题一，绿色=议题二，橙色=议题三……）
- 统一 CSS 变量控制颜色：`--blue` `--teal` `--amber` `--green` `--purple`
- 卡片用 1px border + 10px border-radius，悬停有轻微阴影
- 重要信息用彩色背景 info-box 突出，讨论问题用橙色 info-box

### 页面内容密度
- 每页文字量控制：标题 1 行，副标题 1 行，正文区域不超过页面 70% 高度
- 表格行数不超过 8 行，超出则拆页
- 三列卡片是最大列数，超出改用分组两行布局

### 封面页必含元素
- 项目/组织标签（小标签芯片）
- 主标题（大字，可加颜色高亮关键词）
- 副标题（一句话说明会议目的）
- 议程列表（编号 + 议题名 + 一句话说明）

## 会议文档规范

与 PPT 结构对应，适合会前发送预读或会后存档：
- 用 Markdown 格式
- 每个议题对应一个 `##` 二级标题
- 重要结论加粗，讨论问题用引用块（`>`）或单独列出
- 行动清单用表格（序号 / 行动项 / 时间要求 / 负责人）
- 文档末尾附「核心原则」一段话总结

## 典型页面模式

详见 `references/page-patterns.md`，包含以下常用模式的 HTML 示例：
- 现状对比（左右两列卡片）
- 三层次框架（顶部色块 + 下方细节列）
- 能力分组（上层分类标签 + 下方卡片）
- 时间线（竖线 + 节点）
- 行动清单（表格）
- 讨论引导（橙色 info-box 结尾）
